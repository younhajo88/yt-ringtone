# YouTube Ringtone Webapp Design

## Goal

Create a personal web tool that runs on this Windows PC and lets a phone or PC browser create downloadable 30-second MP3 ringtone files from YouTube search results or YouTube URLs.

The first version focuses on producing the edited MP3 file. Setting the file as a ringtone is handled outside the app by the user on Android, iPhone, or PC.

## Confirmed Scope

- Run a web server on the Windows PC.
- Access the app from PC or phone browsers using `http://PC_IP:PORT`.
- Do not add login, password, account, or secret-link authentication.
- Search YouTube by keyword.
- Accept a direct YouTube URL.
- Show a small list of search results.
- Download selected audio on the PC server.
- Convert audio to MP3.
- Let the user preview the audio in the browser.
- Let the user choose a start time.
- Generate a 30-second MP3 clip.
- Let the browser download the finished ringtone MP3.

## Out Of Scope For Version 1

- Automatically setting Android or iPhone ringtones.
- User accounts or access control.
- Cloud hosting.
- Public deployment.
- Background mobile app behavior.
- Advanced waveform editing.
- Playlists or batch conversion.
- Permanent music library management.

## Legal And Usage Boundary

The app is intended as a personal tool for content the user is allowed to download and transform. The software should not present itself as a public YouTube conversion service.

Because the server has no authentication, it should be treated as exposed if the port is reachable from the internet.

## Recommended Architecture

Use a small local web application:

- Frontend: browser UI served by the local backend.
- Backend: Node.js server on Windows.
- Downloader: `yt-dlp`.
- Audio processing: `ffmpeg`.
- Storage: local project folders for temporary downloads and generated ringtone files.

Node.js is a good fit because the UI and server can live in one project, and the backend can call `yt-dlp` and `ffmpeg` as child processes.

## Main User Flow

1. User opens the app in a browser.
2. User enters either a search phrase or a YouTube URL.
3. If the input is a search phrase, the app returns up to 5 candidate videos.
4. User selects a video.
5. Server downloads the best available audio stream.
6. Server converts the source audio into an app-local MP3 preview file if needed.
7. Browser loads the preview audio.
8. User plays the track and chooses a start time with a slider and numeric input.
9. User clicks "Create 30s MP3".
10. Server creates a 30-second MP3 clip with `ffmpeg`.
11. Browser shows a download button for the generated file.

## UI Design

The first version should be utilitarian and mobile-friendly:

- Single page layout.
- Search or URL input at the top.
- Result list with title, duration, channel, and thumbnail if available.
- Current selected track section.
- Native audio player for preview.
- Start-time slider.
- Start-time numeric input in seconds.
- Generate button.
- Download button after generation.
- Simple status messages for downloading, converting, clipping, and errors.

Waveform editing is intentionally deferred. Native audio playback plus a start-time slider is enough for the first useful version.

## Backend Endpoints

Proposed endpoints:

- `GET /` serves the web UI.
- `POST /api/search` accepts `{ "query": string }` and returns candidate videos.
- `POST /api/prepare` accepts `{ "url": string }`, downloads audio, and returns a local preview track id.
- `GET /api/audio/:id` streams the prepared MP3 preview.
- `POST /api/clip` accepts `{ "id": string, "startSeconds": number }` and creates a 30-second MP3.
- `GET /api/download/:clipId` downloads the generated ringtone file.

All ids should be generated by the app. The client should never be allowed to pass raw filesystem paths.

## Data And File Layout

Use app-owned local folders:

- `data/source/` for downloaded source audio.
- `data/preview/` for converted preview MP3 files.
- `data/clips/` for generated 30-second MP3 files.
- `data/jobs.json` or a lightweight SQLite database later if history is needed.

For version 1, in-memory job metadata plus files on disk is enough. If the server restarts, old generated files can remain downloadable only if a simple manifest is added later.

## Security Constraints Without Login

Since there is no authentication:

- Bind to a configurable host and port.
- Default to a visible warning in startup logs when binding beyond localhost.
- Never expose arbitrary file reads.
- Never accept shell fragments from the browser.
- Validate YouTube URLs and search query length.
- Run only fixed `yt-dlp` and `ffmpeg` commands assembled from validated arguments.
- Store all generated files inside app-owned folders.
- Limit concurrent downloads/clips to avoid accidental overload.
- Add basic file cleanup for old temporary files.

This does not make the app private, but it limits what an unknown visitor can make the server do.

## Error Handling

The app should show friendly errors for:

- `yt-dlp` not installed or not found.
- `ffmpeg` not installed or not found.
- No search results.
- Download failure.
- Unsupported or unavailable video.
- Clip start time beyond the audio duration.
- Server busy.

The backend should return structured JSON errors with short user-facing messages.

## Testing Plan

Version 1 should be tested with:

- Unit tests for URL detection, input validation, filename sanitization, and time validation.
- Backend route tests for invalid input and expected JSON shapes.
- A smoke test that verifies `yt-dlp` and `ffmpeg` are available.
- Manual browser test on the Windows PC.
- Manual phone-browser test over the PC IP address.

Full automated YouTube download tests can be avoided initially because they are slow and network-dependent.

## Implementation Phases

1. Scaffold Node.js web server and static UI.
2. Add environment checks for `yt-dlp` and `ffmpeg`.
3. Implement URL input and audio preparation.
4. Add search by keyword.
5. Add preview playback in the browser.
6. Add start-time selection and 30-second clipping.
7. Add download endpoint.
8. Add cleanup, limits, and focused tests.
9. Test from phone browser using the Windows PC IP address.

## Version 1 Decisions

- Use plain HTML, CSS, and browser JavaScript for the frontend.
- Keep server metadata in memory for version 1.
- Keep generated files on disk, but do not guarantee old download links after server restart.
- Defer waveform editing until after the first usable version.
- Use a configurable port with `8001` as the default.
- Bind to `0.0.0.0` when the user wants phone access from the network.
